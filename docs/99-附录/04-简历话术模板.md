## 二、STAR 法则：项目描述的黄金结构

### 2.1 什么是 STAR 法则？

STAR 是 Situation（背景）、Task（任务）、Action（行动）、Result（结果）四个单词的首字母缩写。它是描述项目经历最清晰、最受面试官认可的框架。

```
STAR 法则结构：
+----------------------------------------------------------+
| S - Situation（背景）                                    |
|     项目是什么？团队规模？你在其中的角色？                |
|                                                          |
| T - Task（任务）                                         |
|     你负责的具体模块或功能是什么？                        |
|                                                          |
| A - Action（行动）                                       |
|     你用了什么技术？遇到了什么难点？如何解决的？          |
|                                                          |
| R - Result（结果）                                       |
|     项目上线了吗？性能提升了多少？学到了什么？            |
+----------------------------------------------------------+
```

### 2.2 STAR 法则示例

**不好的描述**：
> 做了一个员工管理系统，用了 SpringBoot 和 Vue，实现了增删改查功能。

**好的描述（STAR）**：
> **S**：Tlias 是一个企业级人事管理系统，采用前后端分离架构，我作为核心开发负责后端 API 开发和数据库设计。
> **T**：我负责员工管理模块，需要实现多条件分页查询、批量删除、关联工作经历等复杂业务。
> **A**：我使用 MyBatis 动态 SQL 实现多条件组合查询，用 PageHelper 做后端分页；针对员工和工作经历的一对多关系，设计了事务保证数据一致性；用 AOP 记录操作日志便于审计。
> **R**：系统支持 1000+ 员工数据的分页查询响应时间小于 200ms，代码通过 Code Review，掌握了分层架构设计和 RESTful API 规范。

## 四、项目亮点深度话术

### 4.1 分页查询

**问题背景**：
员工表数据量增大后，一次性查询所有数据会导致前端卡顿、后端内存溢出。

**技术方案**：
```
+----------------------------------------------------------+
| 前端：                                                   |
|   - 页码 + 每页条数 传递给后端                           |
|   - 展示总条数和总页数                                   |
|                                                          |
| 后端：                                                   |
|   - PageHelper.startPage(page, pageSize)                 |
|   - 执行 SELECT 查询                                     |
|   - 封装为 PageBean(total, rows) 返回                    |
|                                                          |
| SQL：                                                    |
|   SELECT * FROM emp LIMIT (page-1)*pageSize, pageSize    |
+----------------------------------------------------------+
```

**简历话术**：
> 针对员工列表数据量大的问题，使用 PageHelper 实现后端物理分页。前端传递页码和页大小，后端通过 LIMIT 语句只查询当前页数据，将单次查询数据量从全表降到 10 条，页面加载时间从 3 秒降到 200ms。

**面试延伸**：
- Q：深分页有什么问题？
- A：深分页时 `LIMIT 100000, 10` 需要扫描前 100010 条记录，性能差。解决方案：使用覆盖索引 + 子查询，或基于自增 ID 的范围查询。

### 4.2 AOP 操作日志

**问题背景**：
管理后台需要记录谁、在什么时候、做了什么操作，用于审计和问题追溯。

**技术方案**：
```java
// 1. 自定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Log {
    String value() default "";
}

// 2. AOP 切面拦截
@Aspect
@Component
public class LogAspect {
    @Around("@annotation(log)")
    public Object around(ProceedingJoinPoint joinPoint, Log log) {
        // 记录开始时间
        long start = System.currentTimeMillis();
        
        // 获取方法信息
        String className = joinPoint.getTarget().getClass().getName();
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();
        
        // 执行原方法
        Object result = joinPoint.proceed();
        
        // 记录日志到数据库
        long cost = System.currentTimeMillis() - start;
        saveLog(className, methodName, args, result, cost);
        
        return result;
    }
}

// 3. 在 Controller 方法上使用
@Log("删除员工")
@DeleteMapping("/emps/{ids}")
public Result delete(@PathVariable List<Integer> ids) { ... }
```

**简历话术**：
> 使用自定义注解 + AOP 切面实现操作日志自动记录。在 Controller 方法上添加 `@Log` 注解即可自动记录类名、方法名、参数、返回值和执行耗时，零侵入业务代码。日志数据持久化到 MySQL，支持按时间和操作人查询审计。

**面试延伸**：
- Q：AOP 的底层原理是什么？
- A：Spring AOP 基于动态代理实现。如果目标类实现了接口，使用 JDK 动态代理；如果没有实现接口，使用 CGLIB 生成子类代理。

### 4.3 JWT 登录认证（Version A）

**问题背景**：
前后端分离架构下，传统的 Session 方案需要服务端存储会话状态，不适合分布式部署。

**技术方案**：
```
JWT 认证流程：
+--------+                                    +--------+
|  前端   |  1. 提交用户名密码                  |  后端   |
|        | -------------------------------->  |        |
|        |                                    |        |
|        |  2. 验证通过，返回 JWT Token        |        |
|        | <-------------------------------- |        |
|        |                                    |        |
|        |  3. 后续请求携带 Token（Header）    |        |
|        | -------------------------------->  |        |
|        |                                    |        |
|        |  4. Filter 校验 Token 有效性       |        |
|        |                                    |        |
|        |  5. 校验通过，放行请求              |        |
|        | <-------------------------------- |        |
+--------+                                    +--------+
```

**简历话术**：
> 基于 JWT 实现无状态登录认证。用户登录成功后，服务端生成包含用户 ID 和过期时间的 JWT Token 返回给前端。前端将 Token 存储在 localStorage，后续请求通过 Header 携带。后端使用 Filter 统一拦截校验 Token 的签名和有效期，校验通过后将用户信息存入 ThreadLocal 供业务层使用。

**面试延伸**：
- Q：JWT 和 Session 有什么区别？
- A：JWT 是无状态的，服务端不存储会话信息，Token 自包含用户信息，适合分布式和微服务。Session 是有状态的，服务端需要存储会话数据，适合单体应用。JWT 缺点是 Token 签发后无法主动失效，需要配合黑名单或短有效期 + 刷新机制。

### 4.4 全局异常处理

**问题背景**：
未处理的异常会直接暴露堆栈信息给前端，既不安全也不友好。

**技术方案**：
```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 捕获业务异常
    @ExceptionHandler(BusinessException.class)
    public Result handleBusinessException(BusinessException e) {
        log.error("业务异常：{}", e.getMessage());
        return Result.error(e.getMessage());
    }

    // 捕获参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result handleValidException(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors()
            .stream()
            .map(error -> error.getField() + ":" + error.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return Result.error(msg);
    }

    // 捕获所有其他异常
    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("系统异常：", e);
        return Result.error("系统繁忙，请稍后重试");
    }
}
```

**简历话术**：
> 使用 `@RestControllerAdvice` 实现全局异常处理。针对业务异常、参数校验异常和系统异常分别处理，业务异常返回具体错误信息，系统异常统一返回"系统繁忙"避免暴露内部细节，同时记录完整堆栈日志便于排查。

### 4.5 NL2SQL 智能查询（Version B）

**问题背景**：
非技术人员需要查询数据库数据，但不会写 SQL。

**技术方案**：
```
NL2SQL 流程：
+----------------------------------------------------------+
| 用户输入："查询教研部入职超过2年的员工"                    |
|           |                                              |
|           v                                              |
| Prompt 工程：将表结构 + 用户问题 组装成 Prompt            |
|           |                                              |
|           v                                              |
| LLM 生成 SQL                                             |
|           |                                              |
|           v                                              |
| JSqlParser 安全校验（只允许 SELECT）                      |
|           |                                              |
|           v                                              |
| 执行 SQL 并返回结果                                       |
+----------------------------------------------------------+
```

**简历话术**：
> 实现 NL2SQL 智能查询功能，非技术人员可以用自然语言查询数据库。系统将表结构信息组装成 Prompt 发送给 LLM 生成 SQL，使用 JSqlParser 对生成的 SQL 做 AST 语法解析，只允许 SELECT 语句，禁止 DELETE/DROP 等危险操作，确保数据安全。

### 4.6 RAG 知识库问答（Version B）

**问题背景**：
企业制度文档多，员工查找信息效率低。

**技术方案**：
```
RAG 架构：
+----------------------------------------------------------+
| 文档上传 -> 文本提取（PDFBox/Tika）                       |
|      |                                                   |
|      v                                                   |
| 文本分块（按段落/固定长度）                              |
|      |                                                   |
|      v                                                   |
| Embedding 向量化 -> 存入向量数据库                        |
|      |                                                   |
```

## 五、常见面试题与回答思路

### 5.1 Java 基础

**Q：HashMap 的底层原理是什么？**
> A：HashMap 底层是数组 + 链表 + 红黑树。JDK 1.8 之前是数组 + 链表，1.8 之后当链表长度超过 8 且数组长度超过 64 时，链表转为红黑树。put 时根据 key 的 hash 值计算数组下标，冲突时用链表或红黑树解决。扩容时容量翻倍，重新计算元素位置。

**Q：ConcurrentHashMap 如何保证线程安全？**
> A：JDK 1.7 使用分段锁（Segment），每个 Segment 是一个独立的 HashMap，锁粒度较粗。JDK 1.8 改为 CAS + synchronized，对每个数组位置（桶）加锁，锁粒度更细，并发性能更好。

### 5.2 Spring/SpringBoot

**Q：Spring 的 IOC 和 AOP 是什么？**
> A：IOC（控制反转）是将对象的创建和依赖关系交给 Spring 容器管理，而不是在代码中手动 new。AOP（面向切面编程）是将日志、事务、权限等横切关注点从业务代码中抽离出来，通过代理机制在方法执行前后插入增强逻辑。

**Q：SpringBoot 自动配置的原理是什么？**
> A：SpringBoot 通过 `@EnableAutoConfiguration` 注解实现自动配置。它会读取 `META-INF/spring.factories` 文件中定义的自动配置类，根据条件注解（如 `@ConditionalOnClass`、`@ConditionalOnProperty`）判断是否生效，从而自动装配 Bean。

### 5.3 数据库

**Q：MySQL 的索引底层是什么结构？**
> A：MySQL InnoDB 引擎使用 B+ 树作为索引结构。B+ 树的特点是：所有数据都存储在叶子节点，叶子节点之间通过指针连接形成有序链表，适合范围查询。非叶子节点只存储键值和指针，可以容纳更多索引，树高更低，查询效率更高。

**Q：什么情况下索引会失效？**
> A：常见情况有：对索引列做函数运算、使用 `!=` 或 `<>`、使用 `OR` 条件（部分情况）、字符串不加引号、like 以 `%` 开头、联合索引不满足最左前缀原则。

### 5.4 框架对比

**Q：MyBatis 和 MyBatis-Plus 有什么区别？**
> A：MyBatis-Plus 是在 MyBatis 基础上的增强工具，提供了通用 CRUD 接口（BaseMapper）、条件构造器（QueryWrapper/LambdaQueryWrapper）、分页插件、代码生成器等功能，可以大幅减少样板代码，提升开发效率。

**Q：JWT 和 Session 有什么区别？**
> A：JWT 是无状态的，服务端不存储会话信息，Token 自包含用户信息，适合分布式和前后端分离。Session 是有状态的，服务端需要存储会话数据，通过 Cookie 中的 SessionID 关联，适合传统单体应用。JWT 的优势是服务端无状态、跨域友好；劣势是 Token 签发后难以主动失效。

### 5.5 项目相关

**Q：项目中遇到的最大困难是什么？怎么解决的？**
> A：（示例）在做员工工作经历的一对多关联保存时，遇到事务不一致的问题。员工保存成功但工作经历保存失败时，数据库出现脏数据。我通过 Spring 的 `@Transactional` 注解将两个操作放在同一个事务中，确保要么全部成功要么全部回滚，同时理解了事务的 ACID 特性。

**Q：如果让你优化这个项目，你会怎么做？**
> A：我会从几个方面优化：1）引入 Redis 缓存热点数据（如部门列表），减少数据库查询；2）使用连接池（HikariCP）优化数据库连接管理；3）对慢查询添加索引并优化 SQL；4）引入消息队列处理日志写入，降低接口响应延迟；5）前端使用 CDN 加速静态资源加载。
