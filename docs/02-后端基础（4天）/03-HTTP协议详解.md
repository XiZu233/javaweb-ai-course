# HTTP 协议详解

## 学习目标

学完本节后，你将能够：
- 像阅读一封信一样读懂 HTTP 请求和响应的每一个部分
- 理解 GET 和 POST 的区别，以及什么时候用哪种
- 使用浏览器开发者工具独立分析任何网站的 HTTP 通信
- 完整描述"用户在浏览器点击按钮后，数据是怎么从数据库跑到页面上的"

---

## 核心知识点

### 1. HTTP 是什么——互联网世界的"通用语言"

#### 1.1 生活中的通信协议

想象你要给朋友寄一封信：
- **信封格式**：左上角写邮编，中间写收件人地址，右下角写寄件人
- **内容格式**：开头写称呼，中间写正文，结尾写祝语和署名
- **邮局规则**：贴足邮票、在规定时间投递、按地址分拣

这些都是"协议"——**约定好的通信规则**。

#### 1.2 HTTP 的角色

HTTP（HyperText Transfer Protocol，超文本传输协议）就是**浏览器和服务器之间通信的协议**。

当你打开 `http://localhost:8080/depts` 时：
1. 浏览器说（HTTP 请求）："服务器你好，我想看一下部门列表"
2. 服务器说（HTTP 响应）："好的，这是你要的数据"

#### 1.3 HTTP 的核心特点

**特点一：基于请求-响应模型**
```
浏览器（客户端）              服务器
      |                        |
      |----- HTTP 请求 ------->|
      |                        |
      |<----- HTTP 响应 -------|
      |                        |
```
- 永远是客户端先发请求，服务器再回复响应
- 服务器不会主动给客户端发消息（想做到这个需要 WebSocket）

**特点二：无状态**
```
第一次请求：浏览器 "我要登录"
           服务器 "登录成功，给你一张票（Token）"

第二次请求：浏览器 "我要查数据"
           服务器 "你是谁？我不记得你"
```
- 服务器不会记住之前发生了什么
- 每次请求都是独立的
- 所以需要用 Token/Cookie 来"证明身份"

**特点三：基于 TCP**
- HTTP 建立在 TCP 协议之上（TCP 保证数据可靠传输）
- 默认端口：HTTP 用 80，HTTPS 用 443

### 2. HTTP 请求结构——拆解一封"电子信"

#### 2.1 请求的整体格式

```
POST /api/users HTTP/1.1              ← 第 1 行：请求行（必须）
Host: localhost:8080                  ← 第 2 行起：请求头（多行）
Content-Type: application/json        ←
Authorization: Bearer eyJhbG...       ←
                                      ← 空行（必须，分隔头和体）
{ "name": "张三", "age": 25 }         ← 请求体（可选，POST/PUT 才有）
```

**类比：寄快递**
- **请求行** = 快递单上的基本信息（收件方式 + 地址 + 服务类型）
- **请求头** = 快递单上的附加信息（保价、易碎、指定快递）
- **空行** = 快递单和包裹之间的分隔线
- **请求体** = 包裹里的实际物品

#### 2.2 请求行详解

```
POST /api/users HTTP/1.1
│     │          │
│     │          └── 协议版本：HTTP/1.1（现在主流）
│     │              HTTP/2：支持多路复用，更快
│     │
│     └── 请求路径（URI）：/api/users
│         不包含域名和协议，纯路径部分
│
└── 请求方法：这次请求要做什么动作
```

#### 2.3 请求头详解（常见字段）

```
Host: localhost:8080
# 目标服务器地址。HTTP/1.1 必须包含！
# 因为一个服务器可能托管多个网站，靠 Host 区分

Content-Type: application/json
# 告诉服务器：我发送的请求体是 JSON 格式
# 其他常见值：
#   application/x-www-form-urlencoded  ← 表单提交（键值对）
#   multipart/form-data               ← 上传文件（含二进制）
#   text/plain                        ← 纯文本
#   application/xml                   ← XML 格式

Content-Length: 35
# 请求体的字节长度。服务器用它知道"数据读完了"

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
# 身份认证信息
# Bearer 后面跟的就是 JWT Token
# 服务器拿到后解析，知道"你是谁"

Accept: application/json
# 告诉服务器：我期望你返回 JSON 格式
# 如果服务器返回 XML，浏览器可能处理不了

User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
# 客户端的身份标识
# 服务器可以据此做不同处理（如返回 PC 版或手机版页面）

Cookie: sessionId=abc123; userName=zhangsan
# 浏览器存储的"小纸条"，每次请求自动带上
# 用于记录登录状态、用户偏好等
```

#### 2.4 请求体详解

**GET 请求没有请求体**（参数放在 URL 里）。

**POST/PUT 请求有请求体**，常见格式：

```
【格式 1：JSON】
Content-Type: application/json

{ "name": "张三", "age": 25 }

【格式 2：表单】
Content-Type: application/x-www-form-urlencoded

name=张三&age=25
# 键值对，用 & 连接，特殊字符要 URL 编码

【格式 3：文件上传】
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary
Content-Disposition: form-data; name="file"; filename="avatar.jpg"
Content-Type: image/jpeg

[二进制文件内容]
------WebKitFormBoundary--
```

### 3. HTTP 响应结构——服务器的"回信"

#### 3.1 响应的整体格式

```
HTTP/1.1 200 OK                       ← 第 1 行：状态行
Content-Type: application/json        ← 第 2 行起：响应头
Content-Length: 156                   ←
                                      ← 空行
{ "code": 1, "data": { ... } }        ← 响应体（实际的返回数据）
```

#### 3.2 状态行详解

```
HTTP/1.1 200 OK
│        │   │
│        │   └── 状态描述：对状态码的文字解释
│        │       200 = OK，404 = Not Found，500 = Internal Server Error
│        │
│        └── 状态码：3 位数字，表示请求的结果
│
└── 协议版本
```

#### 3.3 状态码分类——服务器给你的"暗号"

状态码分 5 大类，记住这个规律：

| 范围 | 类别 | 含义 | 记忆口诀 |
|------|------|------|---------|
| 1xx | 信息 | 请求已收到，继续处理 | 很少见 |
| 2xx | 成功 | 请求已成功处理 | "2" = 耶（Yeah） |
| 3xx | 重定向 | 需要进一步操作 | "3" = 转移（Transfer） |
| 4xx | 客户端错误 | 请求有问题 | "4" = 找（客户端）自己的问题 |
| 5xx | 服务器错误 | 服务器内部出错 | "5" = 我（服务器）的问题 |

**常见状态码详解：**

```
┌────────────────────────────────────────────────────────────┐
│  2xx 成功系列                                               │
├────────────────────────────────────────────────────────────┤
│  200 OK                                                    │
│    含义：请求成功，正常返回                                  │
│    场景：GET 查询成功、POST 创建成功                         │
│    响应体：包含请求的数据                                    │
│                                                            │
│  201 Created                                               │
│    含义：资源创建成功                                       │
│    场景：POST 新增用户后返回 201                            │
│    响应头：Location: /users/123（新资源的地址）              │
│                                                            │
│  204 No Content                                            │
│    含义：请求成功，但无返回内容                              │
│    场景：DELETE 删除成功后返回 204                           │
│    响应体：为空                                             │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  3xx 重定向系列                                             │
├────────────────────────────────────────────────────────────┤
│  301 Moved Permanently                                     │
│    含义：资源永久迁移到新地址                                │
│    场景：网站换域名了，老地址自动跳转到新地址                 │
│                                                            │
│  302 Found（临时重定向）                                    │
│    含义：资源临时在其他位置                                  │
│    场景：未登录用户访问需要登录的页面，跳转到登录页            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  4xx 客户端错误系列                                          │
├────────────────────────────────────────────────────────────┤
│  400 Bad Request                                           │
│    含义：请求格式错误                                       │
│    场景：JSON 格式不对、必填字段缺失、参数类型错误            │
│    举例：把字符串传给 int 类型的字段                         │
│                                                            │
│  401 Unauthorized                                          │
│    含义：未认证（没登录）                                    │
│    场景：请求头没有带 Token，或 Token 过期                   │
│    解决：跳转到登录页重新登录                                │
│                                                            │
│  403 Forbidden                                             │
│    含义：已登录，但无权限                                   │
│    场景：普通用户尝试访问管理员接口                          │
│    区别：401 是"你是谁"，403 是"你不配"                      │
│                                                            │
│  404 Not Found                                             │
│    含义：资源不存在                                         │
│    场景：URL 写错了、资源已被删除                            │
│    举例：访问 /users/999，但用户 999 不存在                  │
│                                                            │
│  405 Method Not Allowed                                    │
│    含义：请求方法不允许                                     │
│    场景：用 GET 调用了只支持 POST 的接口                     │
│                                                            │
│  415 Unsupported Media Type                                │
│    含义：请求体的格式服务器不支持                            │
│    场景：Content-Type 发了 XML，但服务器只接受 JSON          │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  5xx 服务器错误系列                                          │
├────────────────────────────────────────────────────────────┤
│  500 Internal Server Error                                 │
│    含义：服务器内部错误                                     │
│    场景：Java 代码抛异常了（NullPointerException 等）        │
│    解决：查看服务器日志，定位问题                            │
│                                                            │
│  502 Bad Gateway                                           │
│    含义：网关/代理服务器从上游收到了无效响应                  │
│    场景：Nginx 代理到后端，但后端没启动或崩溃了              │
│                                                            │
│  503 Service Unavailable                                   │
│    含义：服务器暂时不可用                                   │
│    场景：服务器过载、维护中                                  │
│                                                            │
│  504 Gateway Timeout                                       │
│    含义：网关超时                                           │
│    场景：后端处理太慢，Nginx 等不及了                        │
└────────────────────────────────────────────────────────────┘
```

#### 3.4 响应头详解

```
Content-Type: application/json; charset=UTF-8
# 告诉浏览器：我返回的是 JSON，用 UTF-8 编码
# 浏览器据此决定怎么解析内容

Content-Length: 256
# 响应体的字节长度

Set-Cookie: sessionId=abc123; Path=/; HttpOnly
# 服务器让浏览器保存一个 Cookie
# HttpOnly：禁止 JavaScript 读取，防止 XSS 攻击
# Path=/：整个网站都有效

Location: /users/123
# 配合 301/302 使用，告诉浏览器跳转到哪里

Cache-Control: no-cache
# 缓存控制：不要缓存这个响应，每次都来问我

Access-Control-Allow-Origin: *
# 跨域配置：允许任何域名的网页访问这个接口
```

### 4. HTTP 方法——CRUD 的"动作词汇"

RESTful 风格的 API 设计，就是把 HTTP 方法当成"动词"使用：

| 方法 | 含义 | 是否幂等 | 是否安全 | 对应 SQL | 使用场景 |
|------|------|---------|---------|---------|---------|
| GET | 获取资源 | 是 | 是 | SELECT | 查询列表、查询详情 |
| POST | 创建资源 | 否 | 否 | INSERT | 新增数据 |
| PUT | 更新资源（全量） | 是 | 否 | UPDATE | 修改全部字段 |
| PATCH | 更新资源（部分） | 否 | 否 | UPDATE | 修改部分字段 |
| DELETE | 删除资源 | 是 | 否 | DELETE | 删除数据 |

**关键概念解释：**

- **幂等性**：同一个请求执行多次，结果和一次一样
  - GET /users/1 执行 10 次 → 结果一样（幂等）
  - POST /users 执行 10 次 → 创建了 10 个用户（不幂等）

- **安全性**：请求不会改变服务器上的数据
  - GET 是安全的（只读）
  - POST/PUT/DELETE 不安全（会修改数据）

**本项目的 RESTful 接口设计：**

```
部门管理接口：
┌─────────────┬─────────────────┬──────────────┐
│  操作       │  请求            │  说明        │
├─────────────┼─────────────────┼──────────────┤
│  查询列表   │  GET    /depts   │              │
│  查询详情   │  GET    /depts/1 │ 1 是部门 ID  │
│  新增部门   │  POST   /depts   │ Body 传 JSON │
│  修改部门   │  PUT    /depts   │ Body 传 JSON │
│  删除部门   │  DELETE /depts/1 │ 1 是部门 ID  │
└─────────────┴─────────────────┴──────────────┘

员工管理接口：
┌─────────────┬───────────────────────────────────┐
│  操作       │  请求                              │
├─────────────┼───────────────────────────────────┤
│  分页查询   │  GET /emps?page=1&pageSize=10     │
│  条件查询   │  GET /emps?name=张&gender=1       │
│  新增员工   │  POST /emps                        │
│  修改员工   │  PUT /emps                         │
│  删除员工   │  DELETE /emps/1,2,3               │
└─────────────┴───────────────────────────────────┘
```

**GET 请求传参的两种方式：**

```
方式 1：路径参数（Path Variable）
GET /users/123
→ 适合标识唯一资源，URL 更简洁
→ SpringBoot：@GetMapping("/{id}") + @PathVariable

方式 2：查询参数（Query String）
GET /users?page=1&pageSize=10&name=张三
→ 适合筛选、分页、排序等非标识性参数
→ 多个参数用 & 连接
→ SpringBoot：@RequestParam
```

### 6. 前后端交互完整流程——从点击到数据的"长征"

让我们追踪一个完整的请求链路："用户点击'查询部门列表'按钮"

```
┌─────────────────────────────────────────────────────────────────────────┐
│  第 1 步：用户操作                                                       │
│  ────────────────────                                                   │
│  用户在浏览器点击"查询"按钮                                               │
│  前端 Vue 组件中的 @click="handleQuery" 被触发                           │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 2 步：前端发送请求（Axios）                                          │
│  ─────────────────────────────                                          │
│  request.get('/depts')                                                  │
│  Axios 拦截器自动在请求头中加入 token                                     │
│  生成的 HTTP 请求：                                                      │
│    GET /depts HTTP/1.1                                                  │
│    Host: localhost:8080                                                 │
│    token: eyJhbGciOiJIUzI1NiIs...                                       │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 3 步：浏览器处理                                                     │
│  ────────────────                                                       │
│  检查域名是否相同 → 决定是否触发 CORS 预检                                │
│  组装完整的 HTTP 报文                                                    │
│  通过 TCP 连接发送到服务器                                                │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 4 步：后端接收请求（Tomcat + Spring MVC）                             │
│  ─────────────────────────────────────────                              │
│  Tomcat（内嵌 Web 服务器）监听 8080 端口                                  │
│  收到 HTTP 请求后，交给 DispatcherServlet                                 │
│  DispatcherServlet 根据 URL "/depts" 找到 DeptController.list()           │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 5 步：Filter 拦截处理（版本 A）                                       │
│  ─────────────────────────────────                                      │
│  TokenFilter 拦截到请求                                                  │
│  从请求头中提取 token                                                    │
│  调用 JwtUtils.parseJWT() 验证 Token 是否有效                             │
│  如果无效 → 直接返回 401，不走 Controller                                 │
│  如果有效 → 放行，继续往下执行                                            │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 6 步：Controller 处理                                                │
│  ────────────────────                                                   │
│  @GetMapping("/depts")                                                  │
│  public Result list() {                                                 │
│      return Result.success(deptService.list());                         │
│  }                                                                      │
│  Controller 不处理业务，直接调用 Service                                  │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 7 步：Service 处理业务逻辑                                            │
│  ───────────────────────────                                            │
│  deptService.list() 被调用                                               │
│  Service 可能做一些业务判断（如权限校验、数据过滤）                         │
│  然后调用 deptMapper.list()                                              │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 8 步：Mapper 执行 SQL                                                │
│  ───────────────────────                                                │
│  MyBatis 解析 @Select("SELECT * FROM dept")                              │
│  从数据库连接池（Druid）获取连接                                          │
│  发送 SQL 到 MySQL 执行                                                  │
│  把 ResultSet 结果集映射成 List<Dept>                                    │
│  归还数据库连接                                                          │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 9 步：数据逐层返回                                                    │
│  ────────────────────                                                   │
│  Mapper 返回 List<Dept> 给 Service                                       │
│  Service 返回 List<Dept> 给 Controller                                   │
│  Controller 包装成 Result.success(data)                                  │
│  Spring MVC 把 Result 对象序列化成 JSON                                  │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 10 步：HTTP 响应返回前端                                              │
│  ─────────────────────────                                              │
│  HTTP/1.1 200 OK                                                        │
│  Content-Type: application/json                                         │
│                                                                         │
│  {"code":1,"msg":"success","data":[{"id":1,"name":"研发部"},...]}        │
└─────────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  第 11 步：前端处理响应                                                  │
│  ────────────────────                                                   │
│  Axios 响应拦截器处理：                                                  │
│    if (response.data.code === 1) → 正常，取出 data                       │
│    if (response.data.code === 0) → 报错，显示 msg                        │
│                                                                         │
│  Vue 组件更新数据：tableData = response.data.data                        │
│  页面重新渲染，表格显示部门列表                                            │
└─────────────────────────────────────────────────────────────────────────┘
```

**整个流程耗时：**
- 本地开发环境：50ms ~ 200ms
- 生产环境：30ms ~ 100ms（如果有缓存可能更快）

### 练习 2：用 curl 模拟请求

**目标**：不依赖浏览器，用命令行工具直接发送 HTTP 请求。

**步骤**：

```bash
# 1. GET 请求 - 查询部门列表
curl http://localhost:8080/depts \
  -H "token: 你的jwt-token"

# 2. POST 请求 - 新增部门
curl -X POST http://localhost:8080/depts \
  -H "Content-Type: application/json" \
  -H "token: 你的jwt-token" \
  -d '{"name":"测试部门"}'

# 3. DELETE 请求 - 删除部门
curl -X DELETE http://localhost:8080/depts/5 \
  -H "token: 你的jwt-token"

# 4. 只看响应头（不要响应体）
curl -I http://localhost:8080/depts \
  -H "token: 你的jwt-token"

# 5. 格式化 JSON 输出（Windows 用 PowerShell）
curl http://localhost:8080/depts -H "token: 你的jwt-token" | python -m json.tool
```

## 常见错误排查

### 阶段 1：网络层问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| `无法访问此网站 / 拒绝连接` | 后端服务没启动，或端口不对 | 1. 检查 IDEA 中项目是否已启动<br>2. 检查 `application.yml` 中的 `server.port`<br>3. 检查防火墙是否拦截 |
| `连接超时` | 后端服务在处理中但没响应 | 检查后端是否有死循环或长时间运行的操作 |

### 阶段 2：HTTP 协议层问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| `404 Not Found` | URL 路径错误或请求方法不匹配 | 1. 检查 URL 是否拼写正确<br>2. 检查 Controller 的 `@RequestMapping` 路径<br>3. 检查请求方法（GET/POST）和接口定义是否一致 |
| `400 Bad Request` | 请求参数格式错误 | 1. 检查 JSON 格式是否合法（用 JSON 校验工具）<br>2. 检查字段名是否和实体类属性一致<br>3. 检查 Content-Type 是否为 application/json |
| `401 Unauthorized` | Token 缺失或过期 | 1. 检查请求头中是否有 token<br>2. 检查 token 是否过期（重新登录获取新 token）<br>3. 检查 token 名称是否正确（是 token 还是 Authorization） |
| `403 Forbidden` | 已登录但无权限 | 检查用户角色是否有权访问该接口 |
| `415 Unsupported Media Type` | Content-Type 不匹配 | 确保 POST 请求设置了 `-H "Content-Type: application/json"` |

### 阶段 3：前后端数据交互问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 前端收到数据但页面不显示 | 前端解析逻辑错误 | 检查浏览器控制台（Console）是否有 JS 报错 |
| 后端返回正确但前端提示错误 | Result 包装类解析问题 | 检查前端是否按 `response.data.data` 取值 |
| 中文显示乱码 | 编码不一致 | 1. 数据库、表、字段字符集设为 utf8mb4<br>2. application.yml 中配置 `spring.datasource.url` 添加 `characterEncoding=UTF-8`<br>3. 响应头中设置 `Content-Type: application/json; charset=UTF-8` |

### 阶段 4：跨域问题（CORS）

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| `Access to XMLHttpRequest has been blocked by CORS policy` | 浏览器安全策略阻止跨域请求 | 1. 开发环境：配置前端代理（vue.config.js 或 vite.config.ts）<br>2. 后端配置 CORS 过滤器允许跨域 |

**版本 A 前端代理配置（vue.config.js）：**
```javascript
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'http://localhost:8080',  // 后端地址
        changeOrigin: true,               // 允许跨域
        pathRewrite: { '^/api': '' }      // 把 /api 替换为空
      }
    }
  }
}
```

## 本节小结

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        HTTP 协议核心知识                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  HTTP = 浏览器和服务器之间的通信协议                                      │
│  核心特点：请求-响应模型、无状态、基于 TCP                                │
│                                                                          │
│  请求结构：                                                               │
│    请求行 = 方法 + 路径 + 协议版本                                        │
│    请求头 = Host、Content-Type、Authorization、token...                  │
│    请求体 = POST/PUT 的数据（JSON/表单/文件）                             │
│                                                                          │
│  响应结构：                                                               │
│    状态行 = 协议版本 + 状态码 + 描述                                      │
│    状态码：2xx 成功 | 3xx 重定向 | 4xx 客户端错 | 5xx 服务器错             │
│    响应头 = Content-Type、Set-Cookie...                                  │
│    响应体 = 返回的数据（JSON）                                            │
│                                                                          │
│  RESTful 方法：                                                           │
│    GET    → 查询（幂等、安全）                                            │
│    POST   → 新增（不幂等）                                                │
│    PUT    → 修改（幂等）                                                  │
│    DELETE → 删除（幂等）                                                  │
│                                                                          │
│  前后端交互 11 步：                                                       │
│    点击 → Axios 请求 → 浏览器处理 → Tomcat → Filter → Controller →        │
│    Service → Mapper → MySQL → 逐层返回 → JSON 响应 → 前端更新页面          │
│                                                                          │
│  调试工具：                                                               │
│    浏览器 F12 → Network → XHR 查看接口请求                                │
│    curl 命令行直接发送 HTTP 请求                                          │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```
