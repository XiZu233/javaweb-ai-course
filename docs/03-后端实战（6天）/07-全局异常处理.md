# 全局异常处理

## 学习目标

学完本节后，你将能够：

- 理解为什么需要"统一捕获异常"（而不是每个方法写 try-catch）
- 区分三种异常类型：业务异常、系统异常、参数校验异常
- 自定义业务异常类，在 Service 层有业务冲突时抛出
- 用 `@RestControllerAdvice` + `@ExceptionHandler` 编写全局异常处理器
- 处理参数校验失败、SQL 错误、未知异常等各种场景
- 理解异常处理的优先级匹配规则

---

## 核心知识点

### 1. 为什么需要全局异常处理——"安全网"思想

#### 1.1 没有全局异常处理时

```java
// ❌ 每个 Controller 方法都要写 try-catch，又臭又长
@GetMapping("/{id}")
public Result getById(@PathVariable Integer id) {
    try {
        Emp emp = empService.getById(id);
        if (emp == null) {
            return Result.error("员工不存在");
        }
        return Result.success(emp);
    } catch (BusinessException e) {
        return Result.error(e.getMessage());
    } catch (SQLException e) {
        log.error("数据库错误", e);
        return Result.error("系统繁忙");
    } catch (Exception e) {
        log.error("未知错误", e);
        return Result.error("系统繁忙");
    }
}
```

**三大问题：**

```
┌──────────────────────────────────────────────────────────────┐
│  ① 代码冗余                                                    │
│     5 个接口写 5 遍 try-catch，同样的逻辑重复 N 次             │
│                                                                │
│  ② 容易遗漏                                                    │
│     漏写 catch → 异常层层外抛 → 用户看到 500 + 满屏堆栈          │
│                                                                │
│  ③ 安全风险                                                    │
│     堆栈信息暴露给前端 → 黑客看到类名、路径、数据库语句           │
│     "/api/user/deleteUserById" → 知道了接口命名规律              │
│     "SQL: SELECT password FROM admin..." → 暴露了表结构          │
└──────────────────────────────────────────────────────────────┘
```

#### 1.2 有全局异常处理时

```java
// ✅ Controller 只关心业务，异常交给"安全网"
@GetMapping("/{id}")
public Result getById(@PathVariable Integer id) {
    Emp emp = empService.getById(id);
    return Result.success(emp);   // 什么都不用管，异常自动被捕获
}
```

```
异常发生 → 全局异常处理器统一捕获 → 记录日志 → 返回友好提示
              ↑ 像一张"安全网"
```

### 2. 异常分类——不是所有错误都一个样

#### 2.1 业务异常（BusinessException）

```
┌──────────────────────────────────────────────────────────────┐
│  来源：业务规则校验失败                                         │
│  例子：部门下还有员工，不能删除                                 │
│       用户余额不足，无法购买                                   │
│       手机号已被注册                                           │
│  处理：返回具体错误信息给用户                                   │
│       {"code": 0, "msg": "该部门下存在员工，无法删除"}          │
│  日志：WARN 级别（不是程序 bug，是业务冲突）                    │
└──────────────────────────────────────────────────────────────┘
```

#### 2.2 系统异常（SystemException）

```
┌──────────────────────────────────────────────────────────────┐
│  来源：程序 bug 或外部依赖故障                                  │
│  例子：空指针异常（NullPointerException）                      │
│       SQL 语法错误                                             │
│       Redis 连接超时                                           │
│  处理：返回"系统繁忙，请稍后重试"，**不暴露细节**               │
│       {"code": 0, "msg": "系统繁忙，请稍后重试"}              │
│  日志：ERROR 级别 + 完整堆栈（开发排查用）                      │
└──────────────────────────────────────────────────────────────┘
```

#### 2.3 参数校验异常（ValidationException）

```
┌──────────────────────────────────────────────────────────────┐
│  来源：前端传的参数不合法                                       │
│  例子：手机号格式错误                                          │
│       必填字段为空                                             │
│       日期格式不对                                             │
│  处理：返回哪个字段 + 什么错误                                  │
│       {"code": 0, "msg": "name: 部门名称不能为空"}            │
│  日志：WARN 级别                                               │
└──────────────────────────────────────────────────────────────┘
```

### 3. 自定义业务异常

#### 3.1 为什么要自定义异常

Java 内置的 `RuntimeException` 太笼统。如果所有异常都用它：

```java
throw new RuntimeException("该部门下存在员工，无法删除");
throw new RuntimeException("系统超时");
throw new RuntimeException("参数不合法");
```

无法区分是"业务冲突"还是"系统错误"，全局处理器只能一视同仁。

#### 3.2 创建自定义异常

```java
package com.tlias.exception;

/**
 * 业务异常
 * 当业务规则不满足时抛出（如部门下还有员工）
 */
public class BusinessException extends RuntimeException {

    public BusinessException(String message) {
        super(message);
    }

    public BusinessException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

**说明：**

- 继承 `RuntimeException` → 不用强制在方法签名写 `throws`
- 两个构造方法 → 直接传消息，或传消息 + 原始异常

#### 3.3 在 Service 层抛出

```java
@Service
public class DeptServiceImpl implements DeptService {

    @Autowired
    private DeptMapper deptMapper;
    @Autowired
    private EmpMapper empMapper;

    @Override
    public void delete(Integer id) {
        Long count = empMapper.countByDeptId(id);
        if (count != null && count > 0) {
            throw new BusinessException("该部门下存在员工，无法删除");
        }
        deptMapper.deleteById(id);
    }
}
```

### 4. 全局异常处理器——`@RestControllerAdvice`

#### 4.1 完整实现

```java
package com.tlias.exception;

import com.tlias.pojo.Result;
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.MissingServletRequestParameterException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.util.stream.Collectors;

@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * ① 处理业务异常（最具体）
     */
    @ExceptionHandler(BusinessException.class)
    public Result handleBusinessException(BusinessException e) {
        log.warn("业务异常：{}", e.getMessage());
        return Result.error(e.getMessage());
    }

    /**
     * ② 处理参数校验异常（JSR303 @Valid @Validated）
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result handleValidation(MethodArgumentNotValidException e) {
        // 收集所有字段错误信息
        String message = e.getBindingResult().getFieldErrors().stream()
                .map(error -> error.getField() + ": " + error.getDefaultMessage())
                .collect(Collectors.joining(", "));
        log.warn("参数校验失败：{}", message);
        return Result.error(message);
    }

    /**
     * ③ 处理缺少必填参数
     */
    @ExceptionHandler(MissingServletRequestParameterException.class)
    public Result handleMissingParam(MissingServletRequestParameterException e) {
        log.warn("缺少参数：{}", e.getParameterName());
        return Result.error("缺少必填参数：" + e.getParameterName());
    }

    /**
     * ④ 处理运行时异常
     */
    @ExceptionHandler(RuntimeException.class)
    public Result handleRuntimeException(RuntimeException e) {
        log.error("运行时异常：{}", e.getMessage(), e);
        return Result.error("系统繁忙，请稍后重试");
    }

    /**
     * ⑤ 兜底：处理所有其他异常
     */
    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("未知异常：{}", e.getMessage(), e);
        return Result.error("系统繁忙，请稍后重试");
    }
}
```

#### 4.2 注解详解

| 注解 | 作用 |
|------|------|
| `@RestControllerAdvice` | 标记为全局异常处理类，同时含 `@ResponseBody` 自动返回 JSON |
| `@ExceptionHandler` | 指定捕获的异常类型，匹配到则执行该方法 |
| 多个 `@ExceptionHandler` | 按异常类型的"最近父类"匹配，最具体的优先 |

#### 4.3 异常匹配优先级

```java
@ExceptionHandler(BusinessException.class)   // ① 最优先
@ExceptionHandler(RuntimeException.class)    // ② 次之
@ExceptionHandler(Exception.class)           // ③ 最兜底
```

```
抛出 BusinessException 时的匹配过程：

  BusinessException ──► 是否有 @ExceptionHandler(BusinessException)?   是 → 执行
             │
             ▼ 不匹配
  RuntimeException ──► 是否有 @ExceptionHandler(RuntimeException)?   是 → 执行
             │
             ▼ 不匹配
       Exception ──► 是否有 @ExceptionHandler(Exception)?             是 → 执行
```

**重要：`@ExceptionHandler` 放在同一个类里才有优先级。**

### 5. 参数校验与异常处理联动

#### 5.1 添加校验注解

```java
package com.tlias.pojo;

import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.Size;
import lombok.Data;

@Data
public class Dept {
    private Integer id;

    @NotBlank(message = "部门名称不能为空")
    @Size(max = 50, message = "部门名称不能超过 50 字符")
    private String name;
}
```

#### 5.2 Controller 开启校验

```java
@PostMapping
public Result save(@RequestBody @Validated Dept dept) {
    deptService.save(dept);
    return Result.success();
}
```

- `@Validated` 激活参数校验
- 校验失败自动抛出 `MethodArgumentNotValidException`
- 被全局异常处理器捕获

#### 5.3 校验注解速查表

| 注解 | 说明 | 例子 |
|------|------|------|
| `@NotBlank` | 非空且非空白 | `"  "` 不通过 |
| `@NotEmpty` | 非空（允许空白字符） | `"  "` 通过 |
| `@NotNull` | 不能为 null | 整数类型常用 |
| `@Size(min, max)` | 字符串/集合长度 | `@Size(max=50)` |
| `@Min(value)` | 数字最小值 | `@Min(1)` |
| `@Max(value)` | 数字最大值 | `@Max(100)` |
| `@Pattern(regex)` | 正则匹配 | 手机号格式 |
| `@Email` | 邮箱格式 | |

**注意：** SpringBoot 3 用 `jakarta.validation` 包，SpringBoot 2 用 `javax.validation`。

#### 5.4 引入依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

### 6. 统一的响应封装——Result

#### 6.1 Result 类

```java
package com.tlias.pojo;

import lombok.Data;

@Data
public class Result {
    private Integer code;   // 1 成功，0 失败
    private String msg;
    private Object data;

    public static Result success() {
        return new Result(1, "success", null);
    }

    public static Result success(Object data) {
        return new Result(1, "success", data);
    }

    public static Result error(String msg) {
        return new Result(0, msg, null);
    }

    public static Result error(String msg, Object data) {
        return new Result(0, msg, data);
    }
}
```

#### 6.2 前端统一处理

```javascript
// response 拦截器
request.interceptors.response.use(
    (response) => {
        const res = response.data;
        if (res.code === 0) {
            ElMessage.error(res.msg);
            return Promise.reject(new Error(res.msg));
        }
        return res.data;
    },
    (error) => {
        ElMessage.error('网络错误，请稍后重试');
        return Promise.reject(error);
    }
);
```

## 常见错误排查

### 阶段 1：配置问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 校验注解不生效 | 没引入 validation 依赖 | pom.xml 添加 `spring-boot-starter-validation` |
| 校验不生效（SpringBoot 3） | 用了 javax 包 | 改用 `jakarta.validation` |
| 异常处理器没被扫描 | 包不在启动类同级或子包下 | 放到 `com.tlias.exception` 等扫描路径内 |

### 阶段 2：异常处理逻辑问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 业务异常返回"系统繁忙" | 优先匹配了父类 | 确保 `@ExceptionHandler(BusinessException)` 在 `@ExceptionHandler(RuntimeException)` 上面 |
| 异常信息还是暴露给前端 | 用了 `@ControllerAdvice` | 改用 `@RestControllerAdvice` |
| 异常返回 500 而不是自定义响应 | 处理器抛了异常 | 处理器方法内部也加 try-catch |

### 阶段 3：日志记录问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 日志没有堆栈信息 | `log.error("msg")` 没传异常对象 | 用 `log.error("msg", e)` |
| 业务异常日志太多 | 日志级别不对 | 业务异常用 `log.warn`，系统异常用 `log.error` |
| 日志里的敏感信息暴露 | 把请求体完整打印了 | 过滤掉 password、token 等字段 |

### 阶段 4：前端处理问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 前端收不到错误信息 | 拦截器只处理了 success 情况 | 拦截器里 `if (code === 0)` 也要处理 |
| 前端弹窗提示"[object Object]" | 直接 alert 了对象 | 提取 `res.msg` 再展示 |
| 前端 404 页面变成后端 404 | 后端没返回统一格式 | 再加一个 `NoHandlerFoundException` 处理器 |

## 本节小结

```
┌────────────────────────────────────────────────────────────────────┐
│                    全局异常处理                                      │
├────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  为什么需要：                                                        │
│    • 避免每个接口写 try-catch                                        │
│    • 防止堆栈信息暴露给前端                                           │
│    • 集中记录错误日志                                                 │
│                                                                     │
│  异常分类：                                                          │
│    业务异常   → 返回具体提示    → WARN 日志                          │
│    系统异常   → 返回"系统繁忙"  → ERROR 日志 + 堆栈                  │
│    校验异常   → 返回字段+错误    → WARN 日志                         │
│                                                                     │
│  自定义异常：                                                        │
│    class BusinessException extends RuntimeException                 │
│                                                                     │
│  全局异常处理器：                                                    │
│    @RestControllerAdvice                                            │
│    @ExceptionHandler(异常类型)                                       │
│                                                                     │
│  优先级：                                                            │
│    BusinessException > RuntimeException > Exception（兜底）         │
│                                                                     │
│  参数校验：                                                          │
│    实体加 @NotBlank / @Size 等 → Controller 加 @Validated            │
│    校验失败 → MethodArgumentNotValidException → 全局处理器捕获       │
│                                                                     │
│  安全红线：                                                          │
│    • 系统异常不能暴露详情给前端                                       │
│    • 日志不能打印 token / password                                    │
│    • 异常日志必须打印完整堆栈（ERROR 级别）                           │
│                                                                     │
└────────────────────────────────────────────────────────────────────┘
```
