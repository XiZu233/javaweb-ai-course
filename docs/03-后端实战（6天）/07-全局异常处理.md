# 全局异常处理

## 学习目标

- 理解异常分层处理的思想
- 掌握 `@RestControllerAdvice` 全局异常处理
- 能够编写健壮的错误响应，避免敏感信息泄露

## 核心知识点

### 1. 为什么要全局异常处理

没有全局异常处理时：
- 后端抛出异常，前端收到 500 错误页面或堆栈信息
- 堆栈信息暴露给前端，有安全风险
- 每个 Controller 都要写 try-catch，代码冗余

有了全局异常处理：
- 统一捕获所有异常
- 返回友好的错误提示
- 集中记录错误日志

### 2. 异常分类

**业务异常（BusinessException）**：
- 用户输入不合法
- 业务规则不满足（如部门下有员工不能删除）
- 需要返回具体错误信息给用户

**系统异常（SystemException）**：
- 数据库连接失败
- 空指针异常
- 返回通用提示"系统繁忙"，不暴露细节

### 3. 全局异常处理器

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(RuntimeException.class)
    public Result handleRuntimeException(RuntimeException e) {
        log.error("业务异常：{}", e.getMessage());
        return Result.error(e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("系统异常：{}", e.getMessage(), e);
        return Result.error("系统繁忙，请稍后重试");
    }
}
```

**注解说明**：
- `@RestControllerAdvice`：标记为全局异常处理类，同时包含 `@ResponseBody`
- `@ExceptionHandler`：指定要捕获的异常类型
- 可以定义多个 `@ExceptionHandler`，匹配最具体的异常类型

### 4. 自定义业务异常

```java
public class BusinessException extends RuntimeException {
    public BusinessException(String message) {
        super(message);
    }
}
```

Service 层抛出：

```java
@Override
public void delete(Integer id) {
    Long count = empMapper.countByDeptId(id);
    if (count != null && count > 0) {
        throw new BusinessException("该部门下存在员工，无法删除");
    }
    deptMapper.deleteById(id);
}
```

异常处理器中单独处理：

```java
@ExceptionHandler(BusinessException.class)
public Result handleBusinessException(BusinessException e) {
    log.warn("业务异常：{}", e.getMessage());
    return Result.error(e.getMessage());
}
```

### 5. 异常处理优先级

异常匹配遵循"最近父类"原则：

```java
@ExceptionHandler(BusinessException.class)  // 最具体，优先匹配
public Result handleBusiness(...) { ... }

@ExceptionHandler(RuntimeException.class)   // 次之
public Result handleRuntime(...) { ... }

@ExceptionHandler(Exception.class)          // 最通用，兜底
public Result handleException(...) { ... }
```

### 6. 参数校验异常处理

结合 `@Valid` 和 `@RequestBody` 做参数校验：

```java
@Data
public class Dept {
    @NotBlank(message = "部门名称不能为空")
    private String name;
}
```

```java
@PostMapping
public Result save(@RequestBody @Valid Dept dept) { ... }
```

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public Result handleValidation(MethodArgumentNotValidException e) {
    String message = e.getBindingResult().getFieldErrors().stream()
            .map(error -> error.getField() + ": " + error.getDefaultMessage())
            .collect(Collectors.joining(", "));
    return Result.error(message);
}
```

## 常见错误排查

| 错误现象 | 原因 | 解决方案 |
|---------|------|---------|
| 异常被全局处理器捕获后仍返回 500 | @ExceptionHandler 返回值类型不对 | 确保使用 @RestControllerAdvice 或加 @ResponseBody |
| 自定义异常未被捕获 | 异常类型不匹配 | 检查 @ExceptionHandler 参数类型 |
| 异常信息被吞掉 | 日志未打印完整堆栈 | 使用 log.error("msg", e) 打印完整异常 |

## 本节小结

全局异常处理是后端应用的"安全网"。它将异常分类处理：业务异常返回具体提示，系统异常返回通用提示并记录日志。配合自定义异常，可以让代码更加清晰、错误提示更加友好。
