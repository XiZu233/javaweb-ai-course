# 员工管理分页查询

## 学习目标

- 理解分页查询的原理和必要性
- 掌握 PageHelper 分页插件的使用
- 能够封装分页查询结果返回前端

## 核心知识点

### 1. 为什么要分页

当数据量很大时（如员工表有 10 万条记录），一次性查询全部数据会导致：
- 数据库压力大，查询慢
- 网络传输量大
- 前端渲染卡顿

**分页查询**只返回当前页的数据，大幅提升系统性能和用户体验。

### 2. PageHelper 分页插件

PageHelper 是 MyBatis 生态中广泛使用的分页插件，使用非常简单：

**依赖**：
```xml
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper</artifactId>
    <version>5.3.2</version>
</dependency>
```

**分页原理**：
PageHelper 通过 MyBatis 拦截器，在执行 SQL 前自动插入 `LIMIT` 子句，并额外执行一条 `COUNT` 查询获取总记录数。

### 3. 分页查询实现

**Controller**：

```java
@GetMapping
public Result list(
        @RequestParam(required = false) String name,
        @RequestParam(required = false) Integer gender,
        @RequestParam(required = false) String begin,
        @RequestParam(required = false) String end,
        @RequestParam(defaultValue = "1") Integer page,
        @RequestParam(defaultValue = "10") Integer pageSize) {
    PageInfo<Emp> pageInfo = empService.list(name, gender, begin, end, page, pageSize);
    return Result.success(pageInfo);
}
```

**Service**：

```java
@Override
public PageInfo<Emp> list(String name, Integer gender, String begin, String end,
                          Integer page, Integer pageSize) {
    // 1. 设置分页参数（页码、每页条数）
    PageHelper.startPage(page, pageSize);
    
    // 2. 执行查询（紧跟的第一个查询会被分页）
    List<Emp> list = empMapper.list(name, gender, begin, end);
    
    // 3. 封装分页结果
    return new PageInfo<>(list);
}
```

**返回数据结构**：

```json
{
  "code": 1,
  "msg": "success",
  "data": {
    "total": 85,
    "list": [ /* 当前页10条数据 */ ],
    "pageNum": 1,
    "pageSize": 10,
    "pages": 9,
    "hasNextPage": true,
    "hasPreviousPage": false
  }
}
```

### 4. PageInfo 常用属性

| 属性 | 说明 |
|------|------|
| total | 总记录数 |
| list | 当前页数据列表 |
| pageNum | 当前页码 |
| pageSize | 每页条数 |
| pages | 总页数 |
| hasNextPage | 是否有下一页 |
| hasPreviousPage | 是否有上一页 |

### 5. 注意事项

- `PageHelper.startPage()` 必须紧跟在要分页的查询之前
- 中间不能穿插其他查询，否则会导致非预期分页
- 分页只对紧跟的第一个 `SELECT` 有效

```java
// 正确写法
PageHelper.startPage(page, pageSize);
List<Emp> list = empMapper.list(...);  // 这个查询会被分页

// 错误写法
PageHelper.startPage(page, pageSize);
List<Dept> depts = deptMapper.list();   // 这个被分页了！
List<Emp> emps = empMapper.list(...);   // 这个没有分页
```

## 动手练习

### 练习 1：分页查询调试

1. 在员工表中插入 50 条测试数据
2. 调用 `/emps?page=1&pageSize=10` 观察返回结果
3. 切换页码，观察 `hasNextPage` 和 `hasPreviousPage` 的变化

### 练习 2：分页参数校验

对 page 和 pageSize 做边界校验：
- page < 1 时，默认为 1
- pageSize > 100 时，限制为 100（防止恶意查询过大分页）

## 常见错误排查

| 错误现象 | 原因 | 解决方案 |
|---------|------|---------|
| 分页不生效 | PageHelper.startPage 位置不对 | 必须紧跟在要分页的查询之前 |
| 总记录数为 0 | SQL 查询条件过滤了所有数据 | 检查 WHERE 条件 |
| 返回所有数据 | 未引入 PageHelper 依赖或配置错误 | 确认依赖已引入 |

## 本节小结

分页查询是大表查询的标配。PageHelper 通过拦截器自动处理 LIMIT 和 COUNT，让开发者只需关注业务 SQL。记住：**startPage 紧跟查询**是使用 PageHelper 的唯一法则。
