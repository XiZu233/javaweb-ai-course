# 多表关系与设计

## 学习目标

学完本节后，你将能够：

- 理解为什么要把数据"分散到多张表"（数据库范式）
- 区分三种表关系：一对一、一对多、多对多
- 掌握三种 SQL 连接：INNER JOIN / LEFT JOIN / RIGHT JOIN
- 用 MyBatis 的 `<resultMap>` + `<association>` + `<collection>` 完成对象映射
- 知道"嵌套结果"与"嵌套查询"的区别，避免 N+1 问题
- 能完成 Tlias 系统的三表联查（员工 + 部门 + 工作经历）

## 核心知识点

### 1. 为什么要"分表"——把鸡蛋放在不同篮子

#### 1.1 单表的痛苦

假设把所有信息都塞到一张 `emp` 表里：

```sql
emp(id, name, dept_id, dept_name, dept_phone, dept_address, ...)

第 1 行：1, 张三, 1, 研发部, 010-111, 北京海淀
第 2 行：2, 李四, 1, 研发部, 010-111, 北京海淀
第 3 行：3, 王五, 1, 研发部, 010-111, 北京海淀
... 100 个研发部的员工，每人都重复一份"研发部"信息
```

**三大问题：**

```
┌──────────────────────────────────────────────────────────────┐
│  ① 数据冗余                                                   │
│     相同的部门信息存了 100 份，浪费空间                        │
│                                                               │
│  ② 修改异常                                                   │
│     部门改名"研发部" → "技术中心"，要更新 100 行                │
│     漏掉一行就出现"半新半旧"的脏数据                            │
│                                                               │
│  ③ 删除异常                                                   │
│     部门里所有员工离职 → 部门信息也跟着丢了                    │
└──────────────────────────────────────────────────────────────┘
```

#### 1.2 分表的方案

把信息按"实体"拆成多张表，用"外键"建立联系：

```
┌─────────────────┐         ┌──────────────────────┐
│   dept 部门表    │         │     emp 员工表        │
├─────────────────┤         ├──────────────────────┤
│ id  | name      │◄───────┤ id | name | dept_id  │
│ 1   | 研发部    │  外键   │ 1  | 张三 |  1       │
│ 2   | 市场部    │         │ 2  | 李四 |  1       │
│                 │         │ 3  | 王五 |  2       │
└─────────────────┘         └──────────────────────┘

部门信息只存一份，emp 通过 dept_id 引用部门
```

#### 1.3 三大范式（了解即可）

```
第一范式（1NF）：列不可再分，每个字段都是原子值
第二范式（2NF）：非主键字段完全依赖主键
第三范式（3NF）：非主键字段不能依赖其他非主键字段
```

通俗理解：每张表只描述"一类事物"，相关数据用外键关联。

### 2. 三种表关系

#### 2.1 一对一（1:1）

**定义：** A 表的一行 ↔ B 表的一行（不多不少）

**例子：**

- 员工 ↔ 身份证（一个员工有一张身份证）
- 用户 ↔ 用户详细资料（基础信息表 + 详情表分离）

**实现：** 在任意一边加外键 + 唯一约束

```sql
CREATE TABLE id_card (
    id INT PRIMARY KEY,
    emp_id INT UNIQUE,           -- 唯一约束确保 1:1
    no VARCHAR(18),
    FOREIGN KEY (emp_id) REFERENCES emp(id)
);
```

#### 2.2 一对多（1:N）—— 最常见

**定义：** A 表一行 ↔ B 表多行

**例子：**

- 部门 → 员工（一个部门有多个员工，一个员工只属于一个部门）
- 用户 → 订单（一个用户有多笔订单）
- 班级 → 学生

**实现：** 在"多"的一方加外键

```sql
-- "1"端：部门表
CREATE TABLE dept (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);

-- "N"端：员工表，加外键 dept_id
CREATE TABLE emp (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    dept_id INT,                              -- 外键
    FOREIGN KEY (dept_id) REFERENCES dept(id)
);
```

**记忆：外键永远在"多"的一方。**

#### 2.3 多对多（M:N）

**定义：** A 表多行 ↔ B 表多行

**例子：**

- 学生 ↔ 课程（一个学生选多门课，一门课被多个学生选）
- 用户 ↔ 角色（一个用户有多个角色，一个角色被多个用户拥有）
- 商品 ↔ 标签

**实现：** 必须有"中间表"

```sql
CREATE TABLE student (id INT PRIMARY KEY, name VARCHAR(50));
CREATE TABLE course  (id INT PRIMARY KEY, name VARCHAR(50));

-- 中间表：拆解多对多为两个一对多
CREATE TABLE student_course (
    student_id INT,
    course_id  INT,
    score INT,                            -- 中间表也可以有自己的字段
    PRIMARY KEY (student_id, course_id),  -- 联合主键
    FOREIGN KEY (student_id) REFERENCES student(id),
    FOREIGN KEY (course_id)  REFERENCES course(id)
);
```

#### 2.4 Tlias 系统的关系

```
┌────────────────────────────────────────────────────────────────┐
│                       Tlias 表关系                              │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│   dept (部门)                                                    │
│     │ 1                                                          │
│     │                                                            │
│     │ N         ← 一对多：一个部门有多个员工                       │
│     ▼                                                            │
│   emp (员工)                                                     │
│     │ 1                                                          │
│     │                                                            │
│     │ N         ← 一对多：一个员工有多条工作经历                   │
│     ▼                                                            │
│   emp_expr (工作经历)                                            │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### 3. SQL 连接查询

#### 3.1 INNER JOIN（内连接）

**只返回两边都匹配的数据。**

```sql
SELECT e.name, d.name AS dept_name
FROM emp e
INNER JOIN dept d ON e.dept_id = d.id;
```

**示意图：**

```
   emp                dept
  ┌────┐             ┌────┐
  │张三│ → dept_id=1 │ 1  │研发部
  │李四│ → dept_id=2 │ 2  │市场部
  │王五│ → dept_id=NULL│  │
  └────┘             └────┘

INNER JOIN 结果：
  张三  研发部
  李四  市场部
  （王五没部门，被排除）
```

#### 3.2 LEFT JOIN（左外连接）

**保留左表全部数据，右表不匹配的填 NULL。**

```sql
SELECT e.name, d.name AS dept_name
FROM emp e
LEFT JOIN dept d ON e.dept_id = d.id;
```

**结果：**

```
张三  研发部
李四  市场部
王五  NULL    ← 王五没分配部门，但仍然显示
```

**Tlias 用 LEFT JOIN 的原因：员工可能还没分配部门，但仍要显示在列表里。**

#### 3.3 RIGHT JOIN（右外连接）

**保留右表全部数据，左表不匹配的填 NULL。**

> 实际开发中很少用 RIGHT JOIN，调换两表位置改成 LEFT JOIN 就行。

#### 3.4 三种 JOIN 对比

```
┌───────────────────────────────────────────────────────────────┐
│  连接类型     行为                       什么时候用              │
├───────────────────────────────────────────────────────────────┤
│  INNER JOIN  两边都有才显示              只关心有关联的数据      │
│  LEFT JOIN   左表全显示，右表不匹配填空   主表数据必须显示        │
│  RIGHT JOIN  右表全显示                  少用                   │
└───────────────────────────────────────────────────────────────┘
```

### 4. MyBatis 关联映射——把 SQL 结果"塑形"成对象

#### 4.1 问题：JOIN 出来的扁平结果如何变成嵌套对象？

**SQL 查询结果（扁平）：**

```
┌────┬──────┬─────────┬─────────┐
│ id │ name │ dept_id │dept_name│
├────┼──────┼─────────┼─────────┤
│ 1  │ 张三 │   1     │ 研发部  │
│ 2  │ 李四 │   1     │ 研发部  │
└────┴──────┴─────────┴─────────┘
```

**期望的 Java 对象（嵌套）：**

```java
Emp {
    id: 1,
    name: "张三",
    dept: Dept {
        id: 1,
        name: "研发部"
    }
}
```

需要告诉 MyBatis：**"id 列对应 emp.id，dept_id 列对应 emp.dept.id，dept_name 列对应 emp.dept.name"**。

这就是 `<resultMap>` 的作用。

#### 4.2 resultMap 基础语法

```xml
<resultMap id="myResult" type="com.tlias.pojo.Emp">
    <!-- 主键映射 -->
    <id property="id" column="id"/>

    <!-- 普通字段映射（字段名与列名不一致时必须写） -->
    <result property="deptId" column="dept_id"/>
    <result property="updateTime" column="update_time"/>

    <!-- 一对一关联 -->
    <association property="dept" javaType="com.tlias.pojo.Dept">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>

    <!-- 一对多关联 -->
    <collection property="exprList" ofType="com.tlias.pojo.EmpExpr">
        <id property="id" column="expr_id"/>
        <result property="company" column="company"/>
    </collection>
</resultMap>
```

| 标签 | 作用 |
|------|------|
| `<id>` | 主键映射（必填，用于去重） |
| `<result>` | 普通字段映射 |
| `<association>` | 一对一关联对象 |
| `<collection>` | 一对多关联集合 |

### 5. 一对一映射：`<association>`

#### 5.1 实体类

```java
@Data
public class Emp {
    private Integer id;
    private String name;
    private Integer deptId;
    private Dept dept;       // 一对一关联：员工的部门
}

@Data
public class Dept {
    private Integer id;
    private String name;
}
```

#### 5.2 嵌套结果方式（推荐：一次 SQL 搞定）

```xml
<resultMap id="empWithDept" type="com.tlias.pojo.Emp">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <association property="dept" javaType="com.tlias.pojo.Dept">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>
</resultMap>

<select id="getById" resultMap="empWithDept">
    SELECT e.id, e.name, d.id AS dept_id, d.name AS dept_name
    FROM emp e
    LEFT JOIN dept d ON e.dept_id = d.id
    WHERE e.id = #{id}
</select>
```

**特点：** 一条 SQL 查到所有数据，性能好。

#### 5.3 嵌套查询方式（容易 N+1）

```xml
<resultMap id="empWithDept" type="com.tlias.pojo.Emp">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <!-- 用另一条 SQL 查部门 -->
    <association property="dept"
                 javaType="com.tlias.pojo.Dept"
                 select="com.tlias.mapper.DeptMapper.getById"
                 column="dept_id"/>
</resultMap>

<select id="getById" resultMap="empWithDept">
    SELECT * FROM emp WHERE id = #{id}
</select>
```

**特点：** MyBatis 会先查 emp，再用 dept_id 调用 DeptMapper.getById 查部门。**简单但性能差**（多一次数据库往返）。

### 6. 一对多映射：`<collection>`

#### 6.1 实体类

```java
@Data
public class Emp {
    private Integer id;
    private String name;
    private List<EmpExpr> exprList;   // 一对多：工作经历列表
}

@Data
public class EmpExpr {
    private Integer id;
    private Integer empId;
    private LocalDate begin;
    private LocalDate end;
    private String company;
    private String job;
}
```

#### 6.2 嵌套结果方式

```xml
<resultMap id="empWithExpr" type="com.tlias.pojo.Emp">
    <id property="id" column="id"/>
    <result property="name" column="emp_name"/>
    <collection property="exprList" ofType="com.tlias.pojo.EmpExpr">
        <id property="id" column="expr_id"/>
        <result property="empId" column="emp_id"/>
        <result property="begin" column="begin"/>
        <result property="end" column="end"/>
        <result property="company" column="company"/>
        <result property="job" column="job"/>
    </collection>
</resultMap>

<select id="getById" resultMap="empWithExpr">
    SELECT
        e.id, e.name AS emp_name,
        ee.id AS expr_id, ee.emp_id, ee.begin, ee.end, ee.company, ee.job
    FROM emp e
    LEFT JOIN emp_expr ee ON e.id = ee.emp_id
    WHERE e.id = #{id}
</select>
```

**关键：MyBatis 用 `<id>` 列去重。** 如果员工有 3 条经历，SQL 返回 3 行，但 MyBatis 看到 3 行的 `id` 都是 1，会合并成 1 个 Emp 对象 + 3 条 EmpExpr。

#### 6.3 嵌套查询方式

```xml
<resultMap id="empWithExpr" type="com.tlias.pojo.Emp">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <collection property="exprList"
                ofType="com.tlias.pojo.EmpExpr"
                select="com.tlias.mapper.EmpExprMapper.getByEmpId"
                column="id"/>   <!-- 把 emp.id 作为参数传给子查询 -->
</resultMap>
```

```java
@Mapper
public interface EmpExprMapper {
    @Select("SELECT * FROM emp_expr WHERE emp_id = #{empId}")
    List<EmpExpr> getByEmpId(Integer empId);
}
```

### 7. N+1 问题——嵌套查询的陷阱

#### 7.1 什么是 N+1

```java
// 查询所有员工 + 每人的工作经历
List<Emp> list = empMapper.list();
```

**用嵌套查询时实际发生的 SQL：**

```
SELECT * FROM emp;                                  ← 1 次（查 N 个员工）
SELECT * FROM emp_expr WHERE emp_id = 1;            ← 第 1 次子查询
SELECT * FROM emp_expr WHERE emp_id = 2;            ← 第 2 次
SELECT * FROM emp_expr WHERE emp_id = 3;            ← 第 3 次
...                                                  ← 共 N 次
SELECT * FROM emp_expr WHERE emp_id = 100;          ← 第 N 次

总计：1 + N 次 SQL
100 个员工 → 101 次查询，每次都要走网络 → 极慢
```

#### 7.2 解决方案

**方案 A：改用嵌套结果（一次 JOIN）**

```sql
SELECT e.*, ee.* FROM emp e LEFT JOIN emp_expr ee ON e.id = ee.emp_id;
-- 1 次 SQL 拿到全部数据
```

**方案 B：开启延迟加载（按需触发子查询）**

```yaml
mybatis:
  configuration:
    lazy-loading-enabled: true
    aggressive-lazy-loading: false
```

只有真正访问 `emp.getExprList()` 时才触发子查询，不访问就不查。

**方案 C：手动批量查询（推荐）**

```java
// ① 一次查所有员工
List<Emp> empList = empMapper.list();

// ② 一次查所有相关工作经历
List<Integer> ids = empList.stream().map(Emp::getId).toList();
List<EmpExpr> allExprs = empExprMapper.getByEmpIds(ids);

// ③ 在内存里组装（按 emp_id 分组）
Map<Integer, List<EmpExpr>> map = allExprs.stream()
        .collect(Collectors.groupingBy(EmpExpr::getEmpId));

empList.forEach(e -> e.setExprList(map.get(e.getId())));

// 总计：2 次 SQL，比 N+1 快得多
```

### 8. 完整实战：Tlias 员工详情查询

#### 8.1 需求

`GET /emps/1` → 返回：

- 员工基本信息
- 所属部门名称
- 所有工作经历列表

#### 8.2 数据库

```sql
-- 部门表
CREATE TABLE dept (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL
);

-- 员工表
CREATE TABLE emp (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(20),
    gender TINYINT,
    entrydate DATE,
    dept_id INT,
    create_time DATETIME,
    update_time DATETIME
);

-- 工作经历表
CREATE TABLE emp_expr (
    id INT PRIMARY KEY AUTO_INCREMENT,
    emp_id INT NOT NULL,
    begin DATE,
    end DATE,
    company VARCHAR(50),
    job VARCHAR(50)
);
```

#### 8.3 实体类

```java
@Data
public class Emp {
    private Integer id;
    private String name;
    private Integer gender;
    private LocalDate entrydate;
    private Integer deptId;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;

    // 关联属性
    private Dept dept;                  // 一对一
    private List<EmpExpr> exprList;     // 一对多
}

@Data
public class Dept {
    private Integer id;
    private String name;
}

@Data
public class EmpExpr {
    private Integer id;
    private Integer empId;
    private LocalDate begin;
    private LocalDate end;
    private String company;
    private String job;
}
```

#### 8.4 EmpMapper.xml（完整版）

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.tlias.mapper.EmpMapper">

    <resultMap id="empDetail" type="com.tlias.pojo.Emp">
        <id property="id" column="id"/>
        <result property="name" column="name"/>
        <result property="gender" column="gender"/>
        <result property="entrydate" column="entrydate"/>

        <!-- 一对一：部门 -->
        <association property="dept" javaType="com.tlias.pojo.Dept">
            <id property="id" column="dept_id"/>
            <result property="name" column="dept_name"/>
        </association>

        <!-- 一对多：工作经历 -->
        <collection property="exprList" ofType="com.tlias.pojo.EmpExpr">
            <id property="id" column="expr_id"/>
            <result property="empId" column="emp_id"/>
            <result property="begin" column="expr_begin"/>
            <result property="end" column="expr_end"/>
            <result property="company" column="company"/>
            <result property="job" column="expr_job"/>
        </collection>
    </resultMap>

    <select id="getById" resultMap="empDetail">
        SELECT
            e.id,
            e.name,
            e.gender,
            e.entrydate,
            d.id   AS dept_id,
            d.name AS dept_name,
            ee.id  AS expr_id,
            ee.emp_id,
            ee.begin AS expr_begin,
            ee.end   AS expr_end,
            ee.company,
            ee.job   AS expr_job
        FROM emp e
        LEFT JOIN dept d  ON e.dept_id = d.id
        LEFT JOIN emp_expr ee ON e.id = ee.emp_id
        WHERE e.id = #{id}
        ORDER BY ee.begin
    </select>

</mapper>
```

### 9. 新增员工 + 工作经历（事务）

```java
@Service
public class EmpServiceImpl implements EmpService {

    @Autowired
    private EmpMapper empMapper;
    @Autowired
    private EmpExprMapper empExprMapper;

    @Override
    @Transactional   // 关键：保证原子性
    public void save(Emp emp) {
        // ① 插入员工，回填 id
        empMapper.insert(emp);

        // ② 给每条经历设置 emp_id
        if (emp.getExprList() != null && !emp.getExprList().isEmpty()) {
            emp.getExprList().forEach(expr -> expr.setEmpId(emp.getId()));
            empExprMapper.insertBatch(emp.getExprList());
        }
    }
}
```

如果第二步失败，第一步会回滚（事务详见后续章节）。

## 常见错误排查

### 阶段 1：表设计问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 部门改名要更新 N 行 | 数据冗余 | 拆表，emp 只存 dept_id |
| 中间表数据无法插入 | 没建联合主键 | `PRIMARY KEY (id1, id2)` |
| 外键约束阻止删除 | 子表还有数据 | 先删子表，或在程序层做软关联 |

### 阶段 2：SQL JOIN 问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 员工查不到（没部门） | 用了 INNER JOIN | 改 LEFT JOIN |
| 员工被重复显示 | 一对多 JOIN 后出现笛卡尔积 | 用 GROUP BY 或 resultMap 自动合并 |
| `Column 'name' is ambiguous` | 多表都有同名字段 | 加表别名：`e.name`，并起 alias `AS emp_name` |

### 阶段 3：resultMap 问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| `dept` 始终为 null | `<association>` 字段没映射 | 检查 column 是否存在于 SQL 结果 |
| `exprList` 只有第一条 | 没写 `<id>` | `<collection>` 必须有 `<id>` 才能正确合并 |
| 主表数据被合并多次 | 主表 `<id>` 没写 | resultMap 顶层 `<id>` 必填 |
| 返回 JSON 出现循环引用 | Emp 引用 Dept，Dept 又有 List Emp | Dept 中标 `@JsonIgnore` 或不写反向引用 |

### 阶段 4：性能问题

| 错误现象 | 原因分析 | 解决方案 |
|---------|---------|---------|
| 列表接口慢，DB SQL 数量爆炸 | N+1 查询 | 改嵌套结果或批量查询 |
| 一对多查询返回行数过多 | 笛卡尔积 | 限制每个主表只 JOIN 一个一对多关系 |
| 联表查询锁全表 | 没加索引 | 给外键列建索引 |

## 本节小结

```
┌────────────────────────────────────────────────────────────────────┐
│                    多表关系与设计                                    │
├────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  分表的目的：消除数据冗余，避免修改/删除异常                          │
│                                                                     │
│  三种关系：                                                          │
│    一对一：随便一边加外键 + UNIQUE                                    │
│    一对多：在"多"的一方加外键（最常见）                                │
│    多对多：必须中间表                                                 │
│                                                                     │
│  三种 JOIN：                                                         │
│    INNER JOIN  → 只要交集                                            │
│    LEFT JOIN   → 左表全要（项目最常用）                              │
│    RIGHT JOIN  → 右表全要（少用）                                    │
│                                                                     │
│  MyBatis 关联映射：                                                  │
│    <resultMap>    → 自定义结果集映射                                 │
│    <id>           → 主键（合并去重的依据）                            │
│    <result>       → 普通字段                                         │
│    <association>  → 一对一                                           │
│    <collection>   → 一对多                                           │
│                                                                     │
│  两种实现方式：                                                      │
│    嵌套结果（推荐）→ 一条 SQL + JOIN，性能好                          │
│    嵌套查询      → 多条 SQL，简单但 N+1                              │
│                                                                     │
│  避免 N+1：                                                          │
│    ① 嵌套结果（一次 JOIN）                                            │
│    ② 延迟加载                                                        │
│    ③ 手动批量查询 + 内存组装                                          │
│                                                                     │
│  跨表写操作：                                                        │
│    必须加 @Transactional 保证原子性                                   │
│                                                                     │
└────────────────────────────────────────────────────────────────────┘
```
