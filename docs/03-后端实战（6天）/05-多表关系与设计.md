# 多表关系与设计

## 学习目标

- 理解一对一、一对多、多对多的数据库关系
- 掌握 MyBatis 的关联映射（resultMap + collection/association）
- 能够实现员工-部门-工作经历的联合查询

## 核心知识点

### 1. 数据库表关系类型

**一对一**：如员工和身份证，一个员工对应一个身份证。

**一对多**：如部门对员工，一个部门有多个员工。在多方（员工表）添加外键。

**多对多**：如学生和课程，需要中间表。如 `student_course(student_id, course_id)`。

本项目中的关系：
- 部门（dept）→ 员工（emp）：**一对多**
- 员工（emp）→ 工作经历（emp_expr）：**一对多**

### 2. 关联查询方式

**方式一：单表查询 + 程序组装（简单场景）**

先查主表，再循环查子表。缺点：N+1 次查询，性能差。

**方式二：SQL 关联查询（推荐）**

一条 SQL 通过 JOIN 关联多张表。

```sql
-- 查询员工及其部门名称
SELECT e.*, d.name AS dept_name
FROM emp e
LEFT JOIN dept d ON e.dept_id = d.id
WHERE e.id = 1;
```

### 3. resultMap 映射

当字段名与属性名不一致，或需要映射关联对象时，使用 `resultMap`：

```xml
<resultMap id="empResultMap" type="com.tlias.pojo.Emp">
    <!-- 主键映射 -->
    <id property="id" column="id"/>
    <!-- 普通字段映射 -->
    <result property="deptName" column="dept_name"/>
    <!-- 一对多关联：collection -->
    <collection property="exprList" ofType="com.tlias.pojo.EmpExpr"
                select="com.tlias.mapper.EmpExprMapper.getByEmpId" column="id"/>
</resultMap>
```

**属性说明**：
- `property`：实体类中的属性名
- `column`：SQL 查询结果的列名
- `ofType`：集合中元素的类型
- `select`：指定查询子数据的 SQL（懒加载）
- `column`：传递给子查询的参数列

### 4. 员工 + 工作经历实现

**emp_expr 表**：

```sql
CREATE TABLE emp_expr (
    id INT PRIMARY KEY AUTO_INCREMENT,
    emp_id INT COMMENT '员工ID',
    begin DATE COMMENT '开始时间',
    end DATE COMMENT '结束时间',
    company VARCHAR(50) COMMENT '公司名称',
    job VARCHAR(50) COMMENT '职位'
);
```

**Emp 实体类**：

```java
@Data
public class Emp {
    private Integer id;
    private String username;
    private String name;
    private Integer gender;
    // ... 其他字段
    private String deptName;          // 部门名称（非数据库字段，来自 JOIN）
    private List<EmpExpr> exprList;   // 工作经历列表（一对多）
}
```

**EmpMapper.xml**：

```xml
<resultMap id="empResultMap" type="com.tlias.pojo.Emp">
    <id property="id" column="id"/>
    <result property="deptName" column="dept_name"/>
    <collection property="exprList" ofType="com.tlias.pojo.EmpExpr"
                select="com.tlias.mapper.EmpExprMapper.getByEmpId" column="id"/>
</resultMap>

<select id="getById" resultMap="empResultMap">
    select e.*, d.name as dept_name
    from emp e
    left join dept d on e.dept_id = d.id
    where e.id = #{id}
</select>
```

**EmpExprMapper**：

```java
@Mapper
public interface EmpExprMapper {
    @Select("select * from emp_expr where emp_id = #{empId}")
    List<EmpExpr> getByEmpId(Integer empId);

    @Insert("<script>"
          + "insert into emp_expr(emp_id, begin, end, company, job) values "
          + "<foreach collection='list' item='item' separator=','>"
          + "(#{item.empId}, #{item.begin}, #{item.end}, #{item.company}, #{item.job})"
          + "</foreach>"
          + "</script>")
    void insertBatch(List<EmpExpr> list);

    @Delete("delete from emp_expr where emp_id = #{empId}")
    void deleteByEmpId(Integer empId);
}
```

### 5. 内连接 vs 左连接

| 连接类型 | 说明 | 场景 |
|---------|------|------|
| INNER JOIN | 只返回两表匹配的数据 | 必须有关联数据时 |
| LEFT JOIN | 返回左表全部，右表不匹配为 NULL | 关联数据可能为空时 |

员工查询中，员工可能暂未分配部门，使用 `LEFT JOIN` 更合适：

```sql
-- 即使员工没有部门，也能查出员工信息（dept_name 为 NULL）
SELECT e.*, d.name AS dept_name
FROM emp e
LEFT JOIN dept d ON e.dept_id = d.id;
```

## 常见错误排查

| 错误现象 | 原因 | 解决方案 |
|---------|------|---------|
| deptName 为 null | resultMap 未映射该字段 | 添加 `<result property="deptName" column="dept_name"/>` |
| exprList 为 null | collection 配置错误 | 检查 select 和 column 属性 |
| 子查询 N+1 性能问题 | 每条主数据都触发子查询 | 数据量大时改用 JOIN 一次查询 |

## 本节小结

多表关系是实际项目中的常态。通过 `resultMap` 和 `collection` 可以优雅地映射一对多关系。理解 INNER JOIN 和 LEFT JOIN 的区别，根据业务场景选择合适的连接方式。
